Sourcegraph MCP Guide

Connect your agent to real codebase context instead of local guesswork.

Sourcegraph exposes code search and code intelligence as a Model Context Protocol server. That means an MCP-compatible host can search, read, navigate, inspect history, and run Deep Search against your codebase without stuffing the entire repo into the prompt.

Sourcegraph MCP docs See the implementation section
Quick Start

Two endpoint patterns

  • Public open source: use https://sourcegraph.com/.api/mcp for repositories indexed on Sourcegraph.com search.
  • Private code: use https://your-sourcegraph-instance.com/.api/mcp for your own instance.
  • Authentication: Sourcegraph supports OAuth-capable clients and access-token headers.
  • Deep Search only: use the dedicated /.api/mcp/deepsearch endpoint when you want that specialized tool surface.
How Tool Calling Works

Sourcegraph is the retrieval plane inside the agent loop.

1
Initialize the connection

The host creates an MCP client connection, negotiates capabilities with the Sourcegraph MCP server, and learns which tools are available.

2
List tools before acting

The agent can discover search, navigation, file, history, and Deep Search tools explicitly instead of assuming hidden capabilities.

3
Call only the tools it needs

The agent can run repo discovery, keyword and semantic search, file reads, symbol navigation, history lookups, revision comparison, contributor discovery, and Deep Search as the task demands.

4
Bring the evidence back into the reasoning window

Only the retrieved context enters the model’s working memory. That keeps the prompt tight and the evidence relevant.

JSON-RPC Shape

Representative tool call

tools/call MCP request shape 
{
  "jsonrpc": "2.0",
  "id": 7,
  "method": "tools/call",
  "params": {
    "name": "find_references",
    "arguments": {
      "repo": "github.com/acme/platform",
      "path": "pkg/auth/middleware.go",
      "symbol": "AuthMiddleware"
    }
  }
}

The important architectural point is not the request syntax. It is that the agent can discover and call the capability at the moment it realizes the context is missing.

Client Configs

Host setup patterns for common MCP-compatible tools.

Sourcegraph’s current integrations and MCP docs show direct support for multiple clients. For hosts that only accept stdio MCP servers, use a remote bridge like mcp-remote.

claude_desktop_config.json Bridge remote HTTP MCP through stdio 
{
  "mcpServers": {
    "sourcegraph": {
      "command": "npx",
      "args": [
        "-y",
        "mcp-remote",
        "https://sourcegraph.com/.api/mcp"
      ]
    }
  }
}
.mcp.json Project-scoped or user-scoped remote server 
{
  "mcpServers": {
    "sourcegraph": {
      "type": "http",
      "url": "https://your-sourcegraph-instance.com/.api/mcp",
      "headers": {
        "Authorization": "token YOUR_ACCESS_TOKEN"
      }
    }
  }
}

{
  "mcpServers": {

  }
}

Live Session

How an agent uses a Sourcegraph-backed retrieval layer

The server is not “extra context” bolted onto the prompt. It is the retrieval layer. The agent decides what it needs, calls the right tool, reads the evidence, and only then acts.

Public open-source mode: point a compatible client at https://sourcegraph.com/.api/mcp. That uses the same Sourcegraph search surface available at https://sourcegraph.com/search. For private code, use your own Sourcegraph instance URL instead.

What Sourcegraph Exposes

The complete Sourcegraph MCP tool inventory.

These are the Sourcegraph MCP tools an agent can discover from the server. Deep Search deserves special handling: deepsearch({ question }) launches the long-form research run, and deepsearch_read({ identifier }) reopens that conversation later by token or URL.

Category Tool What it gives the agent
Search nls_search({ query }) Semantic search for natural-language exploration when the agent knows the concept or behavior but not the exact token.
Search keyword_search({ query }) Keyword or regex search using Sourcegraph query syntax for exact search, boolean filters, repo targeting, and revision scoping.
Files read_file({ repo, path, startLine?, endLine?, revision? }) Fetches grounded file contents with optional line ranges and revision pinning so the model reads only the exact evidence it needs.
Repos list_repos({ query, limit?, after?, before? }) Discovers relevant repositories before the agent starts searching or reading, which matters in large multi-repo environments.
Files list_files({ repo, path?, revision? }) Lists files and directories inside a repository so the agent can map structure without guessing paths from thin air.
Navigation go_to_definition({ repo, path, symbol, revision? }) Locates the real symbol definition from a usage site, with revision-aware and cross-repo code intelligence.
Navigation find_references({ repo, path, symbol, revision? }) Finds symbol usage across the codebase so the agent can assess downstream consumers and change blast radius.
History commit_search({ repos, messageTerms?, authors?, contentTerms?, files?, after?, before? }) Searches commit history by message, author, content, file path, and time bounds to recover intent and prior fixes.
History diff_search({ pattern, repos, added?, removed?, author?, after?, before? }) Searches code diffs directly to find where a pattern was added or removed across repositories and time.
History compare_revisions({ repo, base, head, first?, after? }) Compares two revisions to inspect the exact file-level delta between branches, commits, tags, or releases.
Ownership get_contributor_repos({ author, limit?, minCommits? }) Discovers contributor activity by repository, which helps agents infer likely ownership or domain familiarity.
Deep analysis deepsearch({ question }) Runs Sourcegraph Deep Search for complex, multi-step questions that need synthesis across many search, navigation, and history hops.
Deep analysis deepsearch_read({ identifier }) Reads a Deep Search conversation later by URL or token so the host can reopen, inspect, or continue prior research.
