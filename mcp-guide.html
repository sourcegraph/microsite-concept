Sourcegraph MCP

Ask agents to investigate real code, not improvise around missing context.

Sourcegraph MCP turns the codebase into an evidence surface an agent can search, navigate, read, compare, and revisit. The job of the prompt is no longer to stuff everything into context. The job is to aim the investigation.

Grounded answer

Repository cleanup is gated in internal/workspaces/delete.go, which calls CanDeleteWorkspace. References show the same guard reused by admin APIs and the async cleanup worker.

list_repos({ query: "workspace permissions authz", limit: 3 })
nls_search({ query: "where are workspace delete permissions enforced" })
go_to_definition({ repo: "github.com/acme/workspaces", path: "internal/workspaces/handler.go", symbol: "DeleteWorkspace" })
find_references({ repo: "github.com/acme/workspaces", path: "internal/authz/permissions.go", symbol: "CanDeleteWorkspace" })
Why It Matters

Sourcegraph MCP changes the shape of a good request.

Strong MCP pages stay simple: one idea, one contrast, one ask. This page keeps that same logic while expanding it into a complete product narrative.

Without

An agent is boxed into whatever is already in the prompt, the active file, and its priors.

  • It can explain the code it sees, but not the services or references outside the current window.
  • It tends to answer too early because the missing context is invisible.
  • The user asks for output: "explain this," "refactor this," "write a helper."
"Refactor this file."

With

The agent can search across repositories, inspect symbols, read exact files, and pull history before committing to an answer.

  • It can map the surrounding system instead of guessing where the behavior lives.
  • It can show file paths, references, snippets, and diffs as evidence.
  • The user asks for investigation: "find," "trace," "compare," "show the codebase evidence."
"Use Sourcegraph MCP to find where this function is used, then explain the downstream impact."
Tool Surface

Everything the agent can reach through Sourcegraph MCP.

These are the tools available via MCP here. The section below outlines what each tool is for:

Search

nls_search({ query })

Semantic search for natural-language exploration when the user knows the behavior they want to trace but not the exact tokens.

keyword_search({ query })

Keyword or regex search using Sourcegraph query syntax for exact matching, repo filters, and precision work.

Files and Repos

list_repos({ query, limit?, after?, before? })

Find the right repositories before the model starts searching blind across a large organization.

list_files({ repo, path?, revision? })

Map a repository or directory structure without inventing paths from memory.

read_file({ repo, path, startLine?, endLine?, revision? })

Pull only the exact file contents and line ranges that the model needs to reason over.

Navigation

go_to_definition({ repo, path, symbol, revision? })

Jump from a usage site to the actual symbol definition that owns the behavior.

find_references({ repo, path, symbol, revision? })

Trace every downstream usage so the agent can assess impact before changing code.

History

commit_search({ repos, messageTerms?, authors?, contentTerms?, files?, after?, before? })

Recover intent from commit history and narrow the search by author, message, content, file, or time window.

diff_search({ pattern, repos, added?, removed?, author?, after?, before? })

Search code diffs directly when the question is about where a behavior was added, changed, or removed.

compare_revisions({ repo, base, head, first?, after? })

Compare branches, commits, or releases to inspect the precise delta the agent is reasoning about.

Ownership and Deep Analysis

get_contributor_repos({ author, limit?, minCommits? })

Use contributor activity as a clue for likely repository ownership and domain familiarity.

deepsearch({ question })

Run a deeper, multi-step research pass when the task spans repos, history, architecture, and synthesis.

deepsearch_read({ identifier })

Reopen a Deep Search thread later by URL or token so the research can stay inspectable and reusable.

Prompt Rules

Six rules for getting better investigation from MCP-enabled agents.

Each section below has one job: teach the user how to ask for better investigation.

1

Know the available tools before you prompt.

If the user understands the search, navigation, history, and Deep Search surface, they can make a request the agent can actually execute.

  • Use semantic search when the concept is fuzzy.
  • Use keyword search when the token, symbol, or repo filter is known.
  • Use definition, references, and file reads when the task depends on exact code evidence.
2

Tell the agent to use MCP tools explicitly.

Do not assume tool use is automatic. Ask for it directly, especially on investigative work.

Example: "Use Sourcegraph MCP to investigate this issue and show the files you relied on."

3

Ask the agent to gather information first.

Lead with investigation, then analysis. That reduces early confident guesses and creates a visible retrieval trail.

Example: "Find the services that depend on this component, then summarize their relationship."

4

Be specific about the system data you want.

Good prompts name the repo, service, symbol, file path, dependency edge, or time window the agent should inspect.

Example: "Search the last five commit diffs in @zoekt touching queue indexing and explain what changed."

5

Break complex requests into steps.

Ask the agent to retrieve first, synthesize second, and recommend last. Small stages produce better search behavior and easier verification.

6

Ask for codebase evidence in the answer.

Require file paths, snippets, references, search results, or diffs. Evidence is what turns a plausible answer into a trustworthy one.

Example: "Provide the code references and supporting snippets that justify your recommendation."

Investigation Loop

What a grounded session should look like end to end.

The right pattern is incremental retrieval: discover the area, move to exact symbols and files, inspect history only when needed, then answer with evidence.

Realistic Sourcegraph MCP session
User: Use Sourcegraph MCP to investigate why users are still getting billed after a workspace is deleted. Find the repos involved, trace the deletion path, inspect the downstream billing consumer, and check whether the event contract changed recently.

Agent:
I'll investigate this with Sourcegraph MCP first, then summarize the failure path with codebase evidence.

Agent plan:
1. Discover the repos involved.
2. Trace the deletion path.
3. Inspect billing cancellation consumers.
4. Check recent diffs if the event contract changed.

list_repos({ query: "workspace provisioning billing", limit: 5 })
nls_search({ query: "workspace deletion should cancel billing subscription" })
keyword_search({ query: "repo:github.com/acme/provisioning OR repo:github.com/acme/billing \"DeleteWorkspace\" OR \"CancelSubscription\"" })
find_references({ repo: "github.com/acme/provisioning", path: "internal/workspaces/delete.go", symbol: "DeleteWorkspace" })
read_file({ repo: "github.com/acme/billing", path: "internal/subscriptions/consumer.go", startLine: 1, endLine: 180 })
diff_search({ pattern: "DeleteWorkspace", repos: ["github.com/acme/provisioning"], removed: true, after: "2026-01-01" })

What the answer should include

The goal is not just a conclusion. The goal is a conclusion backed by a visible retrieval path.

  • Name the repos and files that matter.
  • Point to the definition and the missing downstream consumer or event.
  • Use history tools only if the current code is not enough to explain the regression.
  • State the fix path in terms of contract restoration, call sites, and tests.
