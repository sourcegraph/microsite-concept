Market Signal

Everyone is racing to build a software factory. Not buying one. Building one

A factory isn’t a tool you pick up off a shelf. It’s infrastructure. It’s process. It’s a system that runs reliably at scale without a human shepherding every unit of output. And increasingly, leading engineering organisations are actually building autonomous coding agents that generate code, review, test, and merge pull requests around the clock.

Stripe

Minions: merges over 1,300 AI generated pull requests every week. Background coding agents that go from a Slack message to a CI-passing PR with no human in the loop. Engineers tag the Minion bot in Slack and walk away. The agent runs in an isolated pre-warmed devbox, executes the task, and opens a PR. Over a thousand of these ship to review every week.

Spotify

Honk: has shipped more than 1,500 agent authored PRs into production. Adapting their Fleet Management system into a platform for running Honk, a background coding agent that performs large scale code migrations from natural language instructions, reducing the need for complex transformation scripts and significantly speeding up migrations.

Ramp

Inspect: internal agent now authors roughly 30% of all merged pull requests. A background agent that runs in sandboxed cloud VMs with full access to their entire development environment: Postgres, Redis, Temporal, RabbitMQ, Sentry, Datadog, LaunchDarkly, Buildkite. Engineers can kick off an Inspect session from Slack, a web UI, or a Chrome extension.

Uber

Minion (separate naming coincidence from Stripe’s): has 84% of developers using agentic tooling monthly. A background agent with full monorepo access. On top of that, they built an internal MCP gateway that exposes several internal endpoints; source code, Jira, Slack, engineering documentation and etc. as a unified context surface for agents.

SOFTWARE FACTORY CAN'T FUNCTION WITHOUT RIGHT CONTEXT

Every serious agent team keeps rediscovering the same failure mode.

01

The failure is usually not “the model is weak.”

The recurring problem is that the agent cannot find the right definitions, callers, history, conventions, or downstream dependencies in time.

02

Prompting humans to hand-carry context does not scale.

If users have to guess the right grep commands or manually paste the relevant files, the factory depends on a human preprocessor, not a reliable context system.

03

Bad retrieval fails quietly.

Too much context wastes reasoning budget. Too little context produces plausible output that feels credible but misses the actual implementation and blast radius.

04

Most benchmarks barely test this layer.

Self-contained coding tasks hide the real retrieval problem. Cross-repository, multi-service work is where weak context surfaces are exposed.

Stephanie Jarmak makes the same point in her benchmark writeup.

Context Hierarchy

A software factory needs more than code access.

Why Factories Stall

Context is not one thing. It is a layered production input. Code, runtime state, and team knowledge each answer a different class of question, and missing any one of them can break the loop.

  • Codebase context answers what exists, where it lives, and what depends on it.
  • Operational context answers what the system is doing right now and where it is failing.
  • Organisational context answers what the team already decided, owns, and expects.
01

Codebase Context

Definitions, references, symbol search, semantic diffs, historical changes, and cross-repository dependency understanding. This is the layer the article connects directly to Sourcegraph.

02

Operational Context

Errors, screenshots, feature flags, queues, deploy state, and observability. This is what lets an agent debug the live system instead of just editing static files.

03

Organisational Context

Jira tickets, Slack threads, ADRs, docs, and ownership rules. This is what makes the resulting code fit the team instead of merely compiling.

A good software factory pattern

Context design should be a foundational consideration in how the software factory operates

Stripe's Minion architecture makes the pattern clear: context is solved before the agent loop starts, not inside it.

Toolshed MCP exposes hundreds of tools across internal systems, docs, tickets, and code intelligence. Before any LLM call, a deterministic orchestrator resolves task references, fetches Jira and docs, runs code search, and hydrates the context window.

That is the pattern: front load the intelligence, assemble context deterministically, then hand the model a narrowed workspace. Sourcegraph adds the code intelligence file access alone cannot: symbols, implementations, callers, deprecations, and cross repository blast radius beyond what grep can reliably reach.

Prehydration Flow
01
Start before the agent loop

Parse the task envelope first: links, repo names, service names, referenced tickets, docs, and the actual problem statement.

02
Resolve organisational context

Fetch Jira, documentation, prior discussion, and ownership signals so the agent knows what change is being requested and why.

03
Resolve codebase context with Sourcegraph

Use semantic search and code intelligence to find the canonical implementation, interface realizations, downstream callers, and deprecated patterns across repositories.

04
Hand off a ranked working set

The model starts from hydrated context instead of exploring a giant filesystem, so it can execute precisely rather than guess what matters.

Does the agent have the correct context?

Can it find the canonical implementation?

If a symbol is implemented in a shared service or hidden behind an interface, the agent should still land on the real source of truth instead of a nearby lookalike.

Good retrieval reduces exploration cost before the model spends tokens reasoning.

Can it trace blast radius across repositories?

The agent should know who calls the thing being changed, which consumers are downstream, and where the migration risk actually sits.

This is where file access alone stops being enough.

Can it bring in history and conventions without flooding the window?

Retrieval quality is precision, not volume. The system should surface the few artifacts that explain prior intent, ownership, and the existing team pattern.

Too much context is another form of failure.

Can it explain the answer with evidence?

A trustworthy output points back to paths, symbols, references, or history. It should feel inspectable, not magical.

The reviewer should be able to verify how the agent knew what it knew.
