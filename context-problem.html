Why context keeps getting skipped

01
Success is invisible
When retrieval works perfectly, reviewers see the PR — not the retrieval mechanism behind it. Precise context gets no credit, so it's never prioritised in evaluations or infrastructure investment.
02
Failure is subtle
Poor context doesn't cause spectacular failures. It produces plausibly correct code that silently diverges — wrong assumptions, stale interfaces, missed conventions.
03
Benchmarks don't test this
Standard LLM evals test isolated, self-contained problems. No benchmark measures cross-repository dependency reasoning at scale.

The silent defect

Context poisoning

Irrelevant context doesn't just dilute signal — it actively corrupts reasoning [1][2]. When 80% of the window is noise, even the correct information gets overridden by what surrounds it.

Ranking matters as much as retrieval [3]. The right answer buried at position 8 of 10 results still produces the wrong code. An agent anchors to what it sees first.

This is also a cost problem [4]. A saturated context window increases latency and token cost on every single agent invocation — at factory scale, this compounds into a meaningful infrastructure burden.

Three layers of context

The architecture every factory needs

Without all three, the agent operates on assumptions.
Each layer is non-negotiable. Together they make output that is genuinely production safe.

Codebase Context
Symbol definitions, cross-repository dependencies, reference navigation, Deepsearch. Not file access — code intelligence. Without it, the agent doesn't know what already exists or how it's connected.
Symbol search Cross-repo refs Deepsearch Ownership
Operational Context
Real-time system state: error traces, feature flags, deploy changes, observability data. Without it, the agent writes code that's theoretically correct but breaks against the live system.
Sentry traces LaunchDarkly Deploy state Metrics
Organisational Context
Team knowledge embedded in JIRA, Slack conversations, architectural decision records, and CODEOWNERS files. Without it, the agent ignores conventions your team has spent years establishing.
JIRA Documentation CODEOWNERS Slack threads
Stripe's Minion · Reference architecture

Context solved before the loop starts

Stripe didn't try to solve context inside the agent loop. They solved it before the loop starts.

Toolshed, their central MCP server, hosts 500 tools covering internal systems, SaaS platforms, docs, tickets, and code intelligence via Sourcegraph. Before any LLM call fires, a deterministic orchestrator scans the task for links and references, pre-fetches relevant context, and curates a focused subset of roughly 15 tools. Tickets fetched. Docs retrieved. Code searched.

By the time the model wakes up, the context window is hydrated and the toolset is scoped. It's not exploring. It's not deciding what to call. It's just doing the work.

Front load the intelligence. Assemble context deterministically. Hand off to the model to execute.

Sourcegraph's role in that architecture

Semantic code intelligence not available through file access alone — which services call this method, where an interface is implemented across repositories, whether a pattern has been deprecated. Grep returns strings. Sourcegraph returns semantically ranked results with cross-repo scope, using the same code intelligence graph that powers the developer-facing search product.
