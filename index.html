How Agents Gather Context

This is not a linear sequence. An agent may plan, retrieve, synthesize, and then return to retrieval several times before acting. The quality of each cycle depends on the context surface available — and retrieval is where most agents fall short.

How MCP Works

An open protocol for pulling context on demand.

Model Context Protocol gives agents a standard way to retrieve information from outside their immediate context window. Without it, the agent is limited to files on the local disk. With it, the agent can call out to any retrieval server — search indexes, code intelligence backends, databases — and pull exactly the evidence it needs, on demand.

Read-only context

Resources expose structured data to the host: schema-like material, documents, or any other context the agent should be able to read without executing an action.

Executable actions

Tools are the workhorses. The agent lists them, then executes focused actions like repo discovery, code search, file reads, symbol navigation, revision comparison, contributor lookup, or Deep Search.

Reusable templates

Prompts package guidance and examples so hosts can reuse proven workflows. They help structure the interaction without overloading every request with boilerplate.

Sourcegraph MCP

How does Sourcegraph MCP work?

  1. 01 Connect

    Your editor connects to Sourcegraph over MCP and negotiates capabilities.

  2. 02 Discover

    The agent lists available tools — search, file reads, definitions, history — before starting.

  3. 03 Retrieve

    It calls only the tools it needs: symbol navigation, references, diffs, Deep Search.

  4. 04 Focus

    Only retrieved evidence enters the reasoning window. No noise, no guesswork.

The agentic loop with Sourcegraph MCP

MCP adds a dedicated retrieval step between discovery and reasoning. Without it, the agent falls back to local files, local history, and local grep. With it, the loop can explicitly pull grounded evidence from a retrieval server before deciding what to do.
